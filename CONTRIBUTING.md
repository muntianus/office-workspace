# Гайд по участию в проекте

Приветствуем всех, кто вносит свой вклад в этот проект! Чтобы поддерживать процесс разработки чистым, эффективным и понятным, мы придерживаемся нескольких ключевых принципов.

## Философия

Мы используем **Trunk-Based Development (TBD)**. Это означает, что `main` является нашей основной и единственной долгоживущей веткой. Все изменения вливаются в `main` через короткоживущие feature-ветки.

Это позволяет нам быстро интегрировать изменения и поддерживать код в состоянии, близком к релизному, в любой момент времени.

## Основной Workflow

1.  **Синхронизация с `main`:**
    Перед началом работы всегда обновляйте вашу локальную `main` ветку.
    ```bash
    git checkout main
    git pull origin main
    ```

2.  **Создание Feature-ветки:**
    Создайте новую ветку от `main`. Название должно отражать суть задачи и следовать формату `<тип>/<краткое-описание>`.
    ```bash
    # Пример для новой фичи
    git checkout -b feat/user-authentication

    # Пример для исправления бага
    git checkout -b fix/incorrect-password-validation
    ```

3.  **Атомарные коммиты:**
    Делайте маленькие, логически завершенные коммиты. Каждый коммит должен представлять собой один шаг в решении задачи.

4.  **Pull Request (PR):
    **Как только ваша задача готова, отправьте ветку в удаленный репозиторий и создайте Pull Request в `main`.
    *   PR должен быть небольшим и сфокусированным на одной задаче.
    *   Название PR и его описание должны следовать правилам [Conventional Commits](#стиль-коммитов).

## Стиль коммитов

Мы придерживаемся спецификации **Conventional Commits**. Это позволяет нам автоматизировать генерацию CHANGELOG и управление версиями.

**Формат:** `<type>(<scope>): <subject>`

*   **type:** Тип изменения. Основные типы:
    *   `feat`: Новая функциональность.
    *   `fix`: Исправление ошибки.
    *   `docs`: Изменения в документации.
    *   `style`: Изменения, не влияющие на логику кода (форматирование, пробелы).
    *   `refactor`: Рефакторинг кода без изменения его функциональности.
    *   `perf`: Улучшение производительности.
    *   `test`: Добавление или исправление тестов.
    *   `build`: Изменения в системе сборки или зависимостях.
    *   `ci`: Изменения в файлах CI/CD.
    *   `chore`: Прочие изменения, не затрагивающие исходный код или тесты.

*   **scope (опционально):** Область кода, которую затронули изменения (например, `auth`, `api`, `ui`).

*   **subject:** Краткое описание изменения в настоящем времени.

**Примеры:**

```
feat(auth): add password reset functionality
fix(api): correct user data validation schema
docs(readme): update installation instructions
```

## Feature Flags

Новая функциональность, которая требует нескольких PR или еще не готова к релизу, должна быть обернута в **Feature Flag**. Это позволяет вливать код в `main` безопасно, не показывая его конечным пользователям до полной готовности.
